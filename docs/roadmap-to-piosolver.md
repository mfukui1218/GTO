# PioSolverレベルへのロードマップ

## 背景

現在のGTO SolverはPreflop〜River全ストリート対応・CFR+/MCCFR実装済みだが、
PioSolverレベルの精度・実用性にはいくつかのギャップがある。

本ドキュメントでは現状分析・PioSolverとの差分・改修計画をまとめる。

---

## 理論的背景: なぜリバーまで考慮が必要か

| 解法 | 特徴 | 問題点 |
|---|---|---|
| フロップまで | フロップ以降のEVをエクイティで近似 | ドローの価値を正確に評価できない。ブラフ/バリューの将来機会を反映不可 |
| リバーまで | 全ストリートの意思決定を織り込んだ真のGTO解 | ゲームツリーが巨大（抽象化が必須） |

**結論**: リバーまで考慮しないと戦略が歪む。現ソルバーはリバーまで対応済み。

### ゲームツリーのサイズと物理限界

- NLHEのフルツリー: **10^160以上のノード**
- 観測可能な宇宙の原子数: 約10^80個
- 完全列挙は物理的に不可能 → **抽象化で10^6〜10^9ノードに圧縮**して解く

### CFRとGTO（ナッシュ均衡）の関係

- CFR/CFR+は2人ゼロサムゲームでナッシュ均衡に**数学的に収束することが証明済み**
- 実用上は「抽象化されたゲームのGTO」であり、元のゲームの厳密なGTOではない
- ただし十分な抽象化精度＋反復数で、人間が搾取不可能なレベルに到達可能

## PioSolverとの差分分析

### PioSolverの主要機能

PioSolver（Pro版）が提供する機能:
- Linear CFR（Discount CFR）による高速収束
- f32精度 + 圧縮によるメモリ効率
- スートイソモーフィズム（等価ボードの同一視）
- ノードロッキング（特定ノードの戦略を固定して再計算）
- サブゲーム解法（ターン/リバーだけを切り出して解く）
- 最大3-4回のレイズ対応
- 任意のベットサイズ数
- GUIによるツリーブラウジング

## 改修計画

### Phase A: メモリ最適化（優先度: 高 / 難易度: 低）

**目的**: 同じマシンで2〜4倍大きなゲームツリーを解けるようにする

#### A-1. f64 → f32 への変換

**対象ファイル**:
- `crates/gto-cfr/src/info_set.rs` — `cumulative_regret`, `cumulative_strategy`: `Vec<f64>` → `Vec<f32>`
- `crates/gto-cfr/src/strategy.rs` — `action_probs`: `Vec<f64>` → `Vec<f32>`
- `crates/gto-cfr/src/cfr.rs` — 各種計算を`f32`に変更

**効果**: メモリ使用量が約1/2に削減。ポーカーの戦略計算ではf32の精度で十分（PioSolverもf32）。

**注意点**:
- exploitability計算など累積値が大きくなる箇所はf64のまま残す選択肢もある
- ベンチマークで収束精度に差がないことを確認

#### A-2. 情報集合キーの最適化

**現状**: `String`キー（例: `"B5|Fx|Tb33x"`）→ ハッシュ＋文字列アロケーションのコスト

**改善案**: 
- 構造化キー（`u64`ビットパッキング）に変更
- バケットID + アクション履歴を固定長バイナリに符号化
- FxHashMapのキー比較が高速化 + メモリ削減

#### A-3. 戦略の圧縮格納

**現状**: 全情報集合で`Vec<f32>`を保持

**改善案**:
- 収束後の最終戦略を量子化（例: 256段階のu8）
- アクションが1つだけの場合は格納不要
- 参照されないノードの戦略を遅延計算

### Phase B: CFRアルゴリズム改善（優先度: 高 / 難易度: 中）

**目的**: 同じ反復数でexploitabilityを桁違いに小さくする

#### B-1. Linear CFR（Discount CFR）の実装

PioSolverが採用しているアルゴリズム。通常のCFR+よりも大幅に速く収束する。

**原理**:
```
反復tでの重み:
  正のリグレット:  (t^α / (t^α + 1)) を掛ける    (α = 1.5)
  負のリグレット:  (t^β / (t^β + 1)) を掛ける    (β = 0)
  累積戦略:        (t / (t + 1))^γ を掛ける       (γ = 2)
```

直近の反復に大きな重みを与えることで、初期の悪い戦略の影響を素早く忘却する。

**実装箇所**:
- `crates/gto-cfr/src/cfr.rs` に `iterate_linear()` を追加
- `TrainerConfig` に `use_linear_cfr: bool` と `alpha`, `beta`, `gamma` パラメータ追加
- 各反復の最後でリグレット・累積戦略にディスカウント係数を乗算

**期待効果**: CFR+と同じ反復数で、exploitabilityが1/10〜1/100に改善

#### B-2. 反復数の動的制御

**現状**: 固定反復数（最大50,000回）

**改善案**:
- exploitabilityの閾値を設定し、達成したら自動停止
- 進捗コールバックでUI側にリアルタイム表示
- 目標精度（例: 0.5% of pot）を指定可能に

### Phase C: スートイソモーフィズム（優先度: 高 / 難易度: 中）

**目的**: 等価なボードを同一視し、計算・メモリを大幅削減

**原理**: 
As♠Kh♥2c♣ と Ad♦Ks♠2h♥ は、スートのラベルが違うだけで戦略的に同一。
スートの置換で同一視できるボードをまとめて1回だけ解く。

**実装方針**:
1. ボードカードのスート正規化関数を実装
2. ハンドのスートも同じ置換で正規化
3. 情報集合キーに正規化後のスートを使用
4. バケッティングでもスート正規化を適用

**効果**: 
- フロップで最大24倍（4!通りのスート置換）の削減（実際は2〜6倍程度）
- ターン/リバーでさらに削減

### Phase D: ノードロッキング（優先度: 高 / 難易度: 中）

**目的**: 実戦的な分析を可能にする（「相手がこう打ったら？」）

**機能仕様**:
- 特定の情報集合の戦略を固定値に設定
- ロックされたノード以外を再計算
- 相手のリーク（例: フロップCBが多すぎる）に対する最適応答を計算

**実装方針**:
1. `InfoSetNode` に `locked: Option<Vec<f32>>` フィールド追加
2. CFR traversalで `locked` がSomeの場合、リグレット更新をスキップし固定戦略を使用
3. Web APIに `locked_nodes: HashMap<String, Vec<f32>>` パラメータ追加
4. UIでノードクリック → 戦略固定 → 再計算のワークフロー

### Phase E: サブゲーム解法（優先度: 中 / 難易度: 高）

**目的**: ターン/リバーの特定スポットだけを高精度で解く

**PioSolverの方式**:
1. フロップを粗い精度で解く
2. 特定のターンカードが出た後のサブツリーだけを再計算
3. フロップ戦略から得られるレンジを入力として使用

**実装方針**:
1. `PostflopGame` にレンジ入力（フロップ計算結果のレンジ）を受け付ける機能追加
2. 任意のストリート・アクション履歴から部分木を構築
3. 部分木のルートに到達確率（レンジ）を設定
4. 部分木だけをCFRで解く

**効果**: 
- フルツリーの1/100以下のメモリで特定スポットを高精度計算
- 実用上最も使われる機能の一つ

### Phase F: UI/UX改善（優先度: 中 / 難易度: 低〜中）

#### F-1. ゲームツリーブラウザ

- 解いたツリーをノード単位でブラウジング
- 各ノードでのレンジ表示・EV表示
- アクションをクリックしてツリーを辿る

#### F-2. 収束グラフ

- 反復数 vs exploitabilityのリアルタイムグラフ
- 目標精度ラインの表示

#### F-3. レンジ比較

- OOP vs IPのレンジを並べて表示
- アクション別のレンジ分割を可視化

## 実装優先順位

```
Phase A: メモリ最適化
  ├── A-1. f32化                    [1-2日]
  ├── A-2. キー最適化               [1日]
  └── A-3. 戦略圧縮                 [1日]

Phase B: CFRアルゴリズム改善
  ├── B-1. Linear CFR実装           [2-3日]
  └── B-2. 動的反復制御             [1日]

Phase C: スートイソモーフィズム       [3-5日]

Phase D: ノードロッキング             [2-3日]

Phase E: サブゲーム解法               [5-7日]

Phase F: UI/UX改善
  ├── F-1. ツリーブラウザ            [3-5日]
  ├── F-2. 収束グラフ               [1日]
  └── F-3. レンジ比較               [1-2日]
```

### 推奨実装順序

1. **A-1 (f32化)** → 即効性が高く、後続の全改修の基盤
2. **B-1 (Linear CFR)** → 収束速度が劇的に改善
3. **D (ノードロッキング)** → 実戦分析の実用性が大幅向上
4. **C (イソモーフィズム)** → メモリ効率の本質的改善
5. **E (サブゲーム)** → PioSolver同等の使い方が可能に
6. **F (UI)** → 使いやすさの仕上げ
